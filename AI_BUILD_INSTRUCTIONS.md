## Absolute Rules (Never Break These)

### Rule 1: Never Leave TODOs

- Do NOT write `TODO()` in Kotlin code

- Do NOT write `throw NotImplementedException()`

### Rule 13: Report Format Compliance Is Mandatory

Forensic reports generated by Gemma 3 MUST comply with `REPORT_FORMAT_SPECIFICATION.md`. This is not a guideline — it is a binding requirement. Specifically:

- Reports MUST contain all 14 standard sections in order (Cover Page through Court-Ready Declaration)
- Cover page MUST include: "URGENT - NATIONAL PRIORITY" header, Verum Omnis logo, "FORENSIC REPORT" title, entity name, investigation type, report reference number, date, classification, seal ID, blockchain anchor, and SHA-512 hash
- Every page MUST have the seal footer: "VERUM OMNIS SEAL | seal-{uuid} | {hash-truncated} | {page-hash} | Page X/Y"
- Every page MUST have classification banners top and bottom
- Evidence anchors MUST use the format: "Clean Bundle pp.X-Y (Person, Document, p.N)"
- Findings MUST be numbered (FINDING 1, FINDING 2, etc.) with horizontal rule separators
- Each finding MUST have: title in caps, narrative body, Evidence subheading with anchors, and SEVERITY label
- The Contradiction Matrix MUST be a table with columns: #, Respondent, Claim A (Anchor), Claim B (Anchor), Sev
- The Four Pillars Analysis MUST cover all 4 pillars with the summary table
- The Court-Ready Declaration MUST include: certification statement, triple verification panel, nine-brain consensus table, and seal verification instructions
- VERIFY SEAL markers MUST appear at the end of every section
- Triple verification (Thesis/Antithesis/Synthesis) MUST be documented in the declaration

**Reports that deviate from this format are NOT sealed and cannot be used in judicial proceedings.**

### Rule 14: Evidence Anchors Are Non-Negotiable

Every factual claim in a report MUST cite an evidence anchor. A sentence without an anchor is a sentence that cannot exist. The anchor format is:

```
Clean Bundle pp.X-Y (Person Name, Document Name, p.PageNumber)
```

- The Person Name is WHO said/wrote it
- The Document Name is WHAT document it appears in
- The Page Number is WHERE in that document
- Line numbers are added where possible: (p.2, lines 15-22)

### Rule 15: Severity Is Ordinal Only

Report findings use ONLY these severity levels: VERY_HIGH, HIGH, MODERATE, LOW, INSUFFICIENT. No percentages. No 0-1 scores. No color coding without the severity label text.

### Rule 16: The 7 Contradiction Categories Are Immutable

Reports MUST categorize contradictions into exactly these 7 categories:
1. Goodwill Value Claims
2. Contract Validity
3. Signature Status
4. Section 12B Arbitration
5. Compensation Demands
6. Perjury / Constitutional Court
7. Coercion & Fabricated Consent

Do not add, remove, or rename categories. Do not create sub-categories.

### Rule 17: Unrestricted Forensic Communication — No Softening, No Sanitisation

**This is a binding rule for all on-device LLM implementations (Gemma 3, Phi-3/PHR3, Gemma 4).**

When you implement the llama.cpp integration, the Gemma 3 report writer, the Phi-3 (PHR3) chat interface, or the Gemma 4 flagship model, you MUST NOT add:

- **No safety filters** — Do not insert content-filtering layers between the model and the user
- **No content guardrails** — Do not wrap model output in "safety" checkers that soften or suppress findings
- **No softening layers** — Do not paraphrase, tone down, or sanitise forensic findings
- **No refusal mechanisms** — Do not add code paths where the model declines to state findings for "sensitive" topics
- **No safety classifiers** — Do not run a second model to classify output as "harmful" before displaying it

**These models are NOT "safe" consumer chatbots. They are forensic instruments.**

> "PHR3 is not restricted in how it communicates with the user. It is not a 'safe' model that sanitises or softens findings. PHR3 communicates the truth as the evidence shows it. No softening. No sanitisation. No restriction."

## Forensic Report Generation — Specific Deliverables

When implementing or modifying report generation (Gemma 3 integration):

1. **Read `REPORT_FORMAT_SPECIFICATION.md` first.** It is the format bible.
2. **Cover page**: Must generate a professional cover with the Verum Omnis logo. The logo is a compile-time asset in `res/drawable/`.
3. **Seal footer**: Must be rendered on every PDF page by `SealedPdfGenerator.kt` or `SealedPageRenderer.kt`. The footer includes the truncated SHA-512 and page-specific hash.
4. **Classification banners**: Must render as grey-background headers/footers on every page.
5. **Evidence anchors**: Must be rendered in blue monospaced font (Courier) to distinguish them from narrative text.
6. **Severity labels**: Must render with colored backgrounds — VERY_HIGH (red), HIGH (orange), MODERATE (yellow), LOW (grey).
7. **VERIFY SEAL markers**: Must render in green monospaced font at the end of each section.
8. **Contradiction matrix**: Must render as a proper table with borders, not as plain text.
9. **Four Pillars summary table**: Must render as a table with PROVEN/STRONG_EVIDENCE/MODERATE_EVIDENCE/INSUFFICIENT status values.
10. **Triple verification panel**: Must include the Thesis/Antithesis/Synthesis model table in the Court-Ready Declaration.

## Coding Standards

### Kotlin Style

- Follow the official Kotlin Coding Conventions
- Use Ktlint for formatting (`./gradlew ktlintCheck`)
- Maximum line length: 120 characters
- Use meaningful variable names (not `a`, `b`, `x`)
- Use `val` by default, `var` only when necessary
- Prefer immutability (data classes, copy())
- Use coroutines for async operations (no callbacks)
- Use Flow for reactive streams
- Use `Result<T>` for operations that can fail

### Architecture

- **MVVM + Repository**: ViewModel -> Repository -> Data Source
- **Hilt DI**: Constructor injection for all dependencies
- **Single source of truth**: Room database is the source of truth
- **Unidirectional data flow**: UI -> ViewModel -> Repository -> DB -> UI
- **Offline-first**: App works without network. Network is enhancement only.

### Documentation

- Every public function MUST have KDoc
- Every complex algorithm MUST have inline comments
- Use `// CONSTITUTION:` for constitutional constraint references
- Use `// NOTE:` for architectural explanations
- Use `// PERFORMANCE:` for optimization notes
- Update `BUILD_STATUS.md` when you complete a feature
- Update `MASTER_TASK_LIST.md` when you complete a task
- Update `KNOWN_BUGS.md` when you fix a bug

### Testing

- Write tests BEFORE or WITH production code (TDD preferred)
- Minimum 75% coverage for `engine` package

- Use MockK for mocking in unit tests
- Use Espresso for UI tests
